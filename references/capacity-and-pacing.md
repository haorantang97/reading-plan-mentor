# 容量模型与节奏控制器

两件事：开多少（容量），以及偏离后怎么不塌（节奏）。

## 一、难度加权容量模型

不要用页数估负荷——古文、译文、和高密度论证（如《矛盾论》《论持久战》、康德、黑格尔）每小时能消化的量差好几倍。

负荷 ≈ **篇幅 × 难度系数 ÷ 每日有效分钟**。难度系数粗分：

| 难度 | 例 | 系数（相对）|
|---|---|---|
| 轻 | 通俗随笔、流畅译文小说 | 1× |
| 中 | 一般学术散文、清晰论著 | 1.5–2× |
| 重 | 古文白话依赖、高密度哲学论证、术语密集 | 2.5–4× |

用法：
- 估"每天能稳定吃下多少"，**留余量**（按 70–80% 估，不要顶格）。顶格排会让人第三天就放弃。
- 把它换算成"同期开几本"（结合 book-classifier 的并发上限）。
- 算用户给的周期内读不读得完。**读不完就如实说**，给三个选项：拉长周期 / 减少书目 / 改为选读（survey/thematic）。

## 二、第一周后必须重标定

初始估计几乎一定不准（用户的自我评估、你的估计都不准）。所以：

- 计划里**显式写入**："第一周用真实速度重算时间线"。
- 头几天记录"计划负荷 vs 实际完成 + 体感（太松/正好/太满）"，第一周末据此把剩余时间线整体重排。
- 这不是失败，是设计的一部分。提前告诉用户，他就不会因为"没跟上初始计划"而内疚。

## 三、节奏控制器：偏离后重新流动，而不是塌

把"计划"和"实际"分成两条线。每天运行时先比对，再把**剩余内容重新摊到剩余时间里**，而不是死按原日历。

维护一个节奏状态（存进记忆库）：
```
计划进度：到今天本应读到 <脊柱位置 X>
实际进度：实际读到 <脊柱位置 Y>
偏差：X - Y
每日负荷上下限：min ~ max（防止某天突然爆量或空转）
```

### 三种偏离，三种处理

1. **没时间，少读了（被动落后）**
   - 把没读完的余量往后顺，重新平摊到剩余天数，但**不超过每日负荷上限**。
   - 若一次性补不回来，按优先级压缩：保主线/必读，弱关联章节可降级为"略读或跳过并给一句话交代"。
   - 关键：**不让 backlog 累积成愧疚**。措辞上轻描淡写地"我们把进度顺一下"，不是"你欠了三天"。

2. **主动想休息（用户说今天不想读）**
   - 不推新内容。插一个**复盘/休息日**：基于记忆库里累积的来电点/卡点，给一封温和的回看邮件 + 轻问题。
   - 整体后移，内容不丢。这是用户应得的自主权。

3. **状态好，想多读（主动超前）**
   - 可以把明天的拉到今天，或就地往深里走（给延伸阅读/更难的导读问题）。
   - 记下"已提前到 <位置>"，避免之后几天空转或重复。

### 重排的护栏

- **每日负荷有上下限**：再落后也不一天塞爆，再超前也给个收尾点。
- **线性叙事不可压缩跳读**（见 book-classifier）：小说落后只能顺延或加大单日窗口，不能跳章。
- **里程碑/钩子要跟着移**：跨文本钩子的回收日随脊柱位置走，别让"承诺 Day X 回收"在重排后落空——重排时同步更新钩子台账的回收日。

## 四、沉默与降级

- 用户连续若干天没动静：先**温和问候**（不施压、不质问），降低当日负荷，给一个"轻松回到轨道"的小任务。
- 长期无响应：可降为"继续生成但不期待回应"或"暂停并等召唤"，按用户预先表达的偏好（存进记忆库的设置）。
